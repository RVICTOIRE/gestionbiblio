Présentation du projet
1.1 Contexte

Le Dakar Institute of Technology (DIT) souhaitait moderniser sa gestion de bibliothèque académique, jusque-là entièrement manuelle. Les problèmes identifiés étaient : le manque de suivi des livres, l'absence de statistiques fiables, une gestion inefficace des emprunts et l'impossibilité pour les étudiants d'accéder au catalogue en ligne.

Ce projet répond à cette demande en développant une plateforme web complète en architecture microservices, dotée d'un système de recommandation de livres basé sur l'apprentissage automatique, tout en démontrant une maîtrise avancée des outils de versioning (Git, DVC, Docker).

1.2 Objectifs atteints
  • ✅ 5 microservices indépendants communicant via API REST
  • ✅ Système de recommandation ML (KNN collaboratif) sur l'historique des emprunts
  • ✅ Versioning avancé avec Git (branches, merges, tags, historique lisible)
  • ✅ Conteneurisation complète avec Docker (multi-stage, non-root, profils dev/prod)
  • ✅ Pipeline de données reproductible avec DVC (preprocess → train → evaluate)
  • ✅ Pipeline CI/CD avec Jenkinsfile (bonus)

2.2 Communication entre services

Les services communiquent uniquement via API REST HTTP sur le réseau interne Docker (bibliotheque-net). Les noms de services Docker servent de DNS interne :

  • Service Emprunts → Service Livres : met à jour la disponibilité lors d'un emprunt/retour via PATCH /books/{id}/availability
  • Service Utilisateurs → Service Emprunts : récupère les statistiques pour le profil via GET /loans/user/{id}/stats
  • Service Recommandation → PostgreSQL directement : lit l'historique des emprunts pour l'entraînement

  preprocess:
    cmd: python ml/preprocess.py
    deps: [ml/preprocess.py, data/loans.csv]
    params: [params.yaml: preprocess]
    outs: [data/loans_clean.csv]

  train:
    cmd: python ml/train.py
    deps: [ml/train.py, data/loans_clean.csv]
    params: [params.yaml: train]
    outs: [model/model.pkl]

  evaluate:
    cmd: python ml/evaluate.py
    deps: [ml/evaluate.py, model/model.pkl, data/loans_clean.csv]
    metrics:
      - metrics.json: {cache: false}
5.3 Rôle de dvc.lock
dvc.lock est le fichier de verrouillage du pipeline DVC. Il enregistre les hash MD5 de chaque fichier d'entrée et de sortie du pipeline. Lors d'un dvc repro, DVC compare les hash actuels avec ceux du dvc.lock : une étape n'est ré-exécutée que si ses dépendances ont changé. Cela garantit la reproductibilité exacte du pipeline à n'importe quel commit Git.
5.4 Détail des étapes ML

preprocess.py — Nettoyage des données :

  • Suppression des doublons et des lignes avec valeurs nulles
  • Calcul du score implicite : retour en avance → 1.5, en retard → 0.8, actif → 1.0
  • Tri et sélection des colonnes finales : user_id, book_id, score, loan_date, status

train.py — Entraînement KNN :

  • Construction d'une matrice creuse user-item (scipy csr_matrix)
  • Entraînement NearestNeighbors(n_neighbors=10, metric='cosine')
  • Sérialisation complète : modèle + matrice + index utilisateurs/livres → model.pkl

evaluate.py — Évaluation :

  • Split 80/20 train/test
  • Prédiction par agrégation pondérée des scores des voisins (poids = 1/distance)
  • Calcul de RMSE, MAE et coverage → metrics.json
5.5 Versioning du modèle

Le fichier model.pkl est versionné par DVC (non committé dans Git). L'API de recommandation charge toujours la bonne version grâce au processus suivant :

# Charger la version v1.0.0 du modèle
git checkout v1.0.0        # Restaure dvc.lock de v1.0.0
dvc checkout               # Télécharge model.pkl correspondant depuis le remote

# Charger la version v1.1.0
git checkout v1.1.0
dvc checkout               # model.pkl mis à jour automatiquement

6.3 Algorithme — Filtrage collaboratif user-based

L'algorithme fonctionne comme suit :

  1. Construction matrice user-item : chaque ligne = utilisateur, chaque colonne = livre, valeur = score implicite
  2. Calcul de similarité cosinus entre utilisateurs via la matrice creuse
  3. Identification des K voisins les plus proches (utilisateurs au profil d'emprunt similaire)
  4. Agrégation : livres empruntés par les voisins mais pas encore par l'utilisateur cible
  5. Ranking : tri par nombre de voisins ayant emprunté le livre
  6. Fallback : si l'utilisateur est inconnu → retourne les livres les plus populaires
6.4 Hyperparamètres (params.yaml)
train:
  n_neighbors: 10      # Nombre de voisins KNN
  metric: cosine       # Métrique de distance

preprocess:
  min_interactions: 1  # Interactions minimum par utilisateur
  test_size: 0.2       # Proportion données de test
